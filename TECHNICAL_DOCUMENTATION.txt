SECTION 1 — PROJECT OVERVIEW
================================================================================

What This Application Is
------------------------
SmileCare Pro is a single-page web application (SPA) built with React and
TypeScript that provides dental clinic operations management and a public
patient booking portal. The product is branded as enterprise dental clinic
management software and ships in portfolio demo mode by default: data persists
in the browser via localStorage, and staff authentication is simulated without
passwords.

The application can optionally connect to Firebase Firestore when
VITE_USE_FIRESTORE=true and Firebase environment variables are configured.

Business Problem It Solves
--------------------------
Dental clinics manage overlapping operational concerns: appointment scheduling,
doctor availability, patient records, service catalogs, payment tracking,
waitlists, and inbound patient inquiries. SmileCare Pro consolidates these
into one web-based platform with:

  - An internal staff dashboard for day-to-day clinic operations
  - A public booking portal (/book/:slug) for online appointment requests
  - A marketing landing page (/) for patient-facing discovery

Target Users
------------
  Patients (public)       — Browse the landing page and book appointments
                            online without login
  Clinic owners / admins  — Configure clinic settings, view reports, manage
                            integrations
  Receptionists           — Manage appointments, patients, waitlist, and
                            inquiries
  Doctors / staff         — View schedules and clinic data (branch-scoped by
                            default)
  System administrators   — Highest permission tier in the demo role selector
                            (demo)

There is NO patient login account system implemented. Patients interact only
through the public website and booking wizard.

Why a Dental Clinic Would Use This Software
-------------------------------------------
The codebase is explicitly tailored to dental workflows:

  - Domain language uses Patients, Doctors, and Dental Services
  - Demo seed data includes dental services (cleaning, root canal, whitening,
    implants, orthodontics)
  - Doctor scheduling includes weekly availability, lunch breaks, and
    conflict detection
  - Patient profiles support clinical notes, tags (e.g., VIP, Orthodontics),
    and visit history

How the System Improves Daily Clinic Operations
------------------------------------------------
  Scheduling        — Week-view calendar with create/edit/delete, conflict
                      warnings, duplication, and status workflow
  Front desk        — Centralized patient database with search, tags, and
                      profiles
  Capacity mgmt     — Waitlist queue with promotion to appointments
  Lead handling     — Kanban-style inquiry pipeline with stage progression
  Visibility        — Dashboard KPIs, notifications, and audit activity log
  Self-service      — Public booking portal reduces phone booking load
  Multi-location    — Branch selector filters data across clinic locations

Main Platform Objectives
------------------------
  1. Provide a unified admin interface for dental clinic operations
  2. Enable online patient self-booking with validation and payment simulation
  3. Support multi-branch clinic data scoping
  4. Offer bilingual (English/Arabic) staff UI with RTL layout
  5. Demonstrate SaaS architecture via repository pattern with mock and
     Firestore backends
  6. Ship as an installable Progressive Web App (PWA)

2.1 Landing Website
-------------------
Route: / (guest-only; redirects logged-in staff to dashboard)

Purpose: Patient-facing marketing site to promote the clinic and drive online
bookings.

How it works:
  - Fixed navigation with anchor links (About, Services, Doctors, Reviews,
    FAQ, Contact)
  - Hero section with CTA to /book/smilecare-clinic
  - Static service cards, doctor profiles, testimonials, FAQ accordion, and
    contact form (toast confirmation only — no backend submission)
  - Links to staff login (/login) and booking portal

Users: Public visitors, prospective patients
Business value: Brand presence, service discovery, conversion to online booking


2.2 Staff Authentication & Login
--------------------------------
Route: /login

Purpose: Gate access to the admin dashboard.

How it works:
  - Email input (default: admin@smilecare.com)
  - Role selector: Super Admin, Clinic Owner, Receptionist, Doctor / Staff
  - No password validation in demo mode
  - Session stored in localStorage (smilecare_auth_session)
  - "Reset Demo Data" reloads dental preset via mockDbInstance.resetPreset('Dental')
  - "Forgot password" button is disabled

Users: Clinic staff
Business value: Role-based access demonstration for portfolio and stakeholder demos


2.3 Dashboard
-------------
Route: /admin/dashboard

Purpose: Operational overview for clinic staff.

How it works:
  - Computes KPIs from reservation and patient data (filtered by selected branch):
      * Today's appointments count
      * Total revenue (Completed or Paid appointments)
      * Patient retention rate (returning patients with multiple completed visits)
      * Cancellation rate
  - Today's schedule table
  - Recent activity from audit logs
  - Upcoming appointments list
  - Quick navigation to services, doctors, waitlist, and public portal

Users: All authenticated staff roles
Business value: At-a-glance clinic health and same-day schedule visibility


2.4 Appointments Calendar (Smart Calendar)
------------------------------------------
Route: /admin/calendar

Purpose: Visual scheduling and appointment lifecycle management.

How it works:
  - Week view: 7-column grid with hourly rows (09:00–16:00); click cells to
    create appointments
  - Day/Month views: Placeholder UI prompting user to switch to Week view
  - Staff filter dropdown
  - Appointment modal supports:
      * Create, edit, delete
      * Status: Pending, Confirmed, Completed, Cancelled, NoShow
      * Payment status: Unpaid, Authorized, Paid, Refunded
      * Payment methods: Cash, Stripe, PayPal, Card
      * Real-time conflict detection via db.reservations.checkConflict()
      * Duplicate appointment (creates copy for next day)
  - On create: dispatches EnterpriseNotificationHub.dispatchBookingAlert()
    (stub notifications)
  - On delete: creates waitlist promotion notification if matching waitlist
    entry exists

Users: Receptionists, owners, doctors (view/manage)
Business value: Prevents double-booking, centralizes appointment workflow


2.5 Patient Management (Customer Database)
------------------------------------------
Route: /admin/customers

Purpose: Maintain patient records.

How it works:
  - List view with search (name, phone, email) and tag filter
  - Create patient modal (name, phone, email, notes, comma-separated tags,
    status)
  - Delete with confirmation
  - Drill-down to Patient Profile showing:
      * Contact info, total spending, booking count
      * Upcoming and past appointments
      * Clinical notes timeline (append-only notes with date stamps)
      * Tags display

Users: Receptionists, owners, doctors
Business value: Single source of truth for patient identity, history, and notes


2.6 Doctor Management (Employee Management)
-------------------------------------------
Route: /admin/employees

Purpose: Manage doctors/staff profiles and weekly availability.

How it works:
  - Card grid showing doctor name, role, contact, branch assignments, completed
    booking metrics
  - Create/edit modal with two tabs:
      * Profile: name, role, email, phone, status, branch checkboxes
      * Schedule: per-day working toggle, start/end times (Mon–Fri default
        09:00–17:00 with 12:00–13:00 break)
  - Delete (labeled "deactivate") removes doctor record

Users: Owners, receptionists
Business value: Availability drives booking slot generation and conflict prevention


2.7 Dental Services Management
------------------------------
Route: /admin/services

Purpose: Maintain service catalog with pricing and duration.

How it works:
  - Service cards showing name, category, duration, price, branch count, status
  - Create/edit modal: name, description, duration, price, category, status,
    branch assignment
  - Delete with confirmation

Users: Owners, receptionists
Business value: Standardized offerings for scheduling, reporting, and public booking


2.8 Waitlist Queue
------------------
Route: /admin/waitlist

Purpose: Track patients waiting for appointment slots.

How it works:
  - Lists Waiting entries filtered by branch
  - Add to waitlist: patient, service, optional preferred doctor, date, time
    preference (Morning/Afternoon/Evening/Any), notes
  - Promote: Creates a Pending appointment at 09:00 and marks waitlist as
    Promoted
  - Cancel: Sets status to Cancelled

Users: Receptionists, owners
Business value: Captures demand when calendar is full; supports slot-fill workflow


2.9 Patient Inquiries Pipeline (Leads)
-------------------------------------
Route: /admin/leads

Purpose: CRM-style lead tracking from inquiry to conversion.

How it works:
  - Kanban board with stages: New -> Contacted -> Interested -> Converted -> Lost
  - Pipeline stats: total value, active deals, hardcoded "+24% month-on-month"
    acquisition rate
  - Add lead modal: name, phone, email, source, estimated value, notes
  - Stage advancement buttons on each lead card

Users: Receptionists, owners
Business value: Structured follow-up on prospective patients


2.10 Reports & Analytics
------------------------
Route: /admin/reports (BusinessOwner+ only)

Purpose: Financial and operational reporting.

How it works:
  - Date range filters: Today, Week, Month, All Time
  - Branch-scoped metrics:
      * Gross booking sales
      * Completion rate
      * Total bookings
      * Cancellations / No-shows
  - Service revenue contribution (progress bars)
  - Doctor revenue leaderboard
  - Export CSV: downloads appointment data
  - Print / PDF: triggers window.print()

Users: Clinic owners, super admins
Business value: Data-driven decisions on services, doctors, and booking performance

NOTE: Translation keys for "30-Day Revenue Forecast" exist but no forecast charts
are rendered in the UI. No charting library is installed.


2.11 Settings
-------------
Route: /admin/settings (BusinessOwner+ only)

Purpose: Clinic configuration and workspace management.

Four tabs:
  Clinic Profile    — Business name, address, phone, email, currency
                      (USD/AED/SAR/EGP), language (en/ar)
  Demo Data Presets — Load dental demo dataset or reset to empty workspace
  Integrations      — Toggle flags: Stripe, PayPal, Google Calendar, WhatsApp,
                      Telegram (stored in settings; most are not enforced in
                      UI flows)
  Activity Log      — Read-only audit log display

Users: Clinic owners, super admins
Business value: Clinic branding, localization, demo control, integration readiness


2.12 Public Booking Portal
--------------------------
Route: /book/:slug (default slug: smilecare-clinic)

Purpose: Patient self-service appointment booking.

4-step wizard:
  1. Select Details: branch, service, doctor, date, available time slots
  2. Patient Information: name, phone, email, notes
  3. Confirm & Pay: Cash / Stripe / PayPal (simulated)
  4. Confirmation: receipt with booking reference

Validation:
  - Past dates blocked
  - Doctor availability checked against weekly schedule and breaks
  - Slot conflict checked client-side and server-side via repository
  - Email and phone format validation

On success:
  - Creates new patient record (tagged Portal-Booking)
  - Creates Pending reservation
  - Creates staff notification

Invalid slug: Shows BusinessNotFound component

Users: Public patients (no authentication)
Business value: 24/7 booking availability, reduced front-desk workload


2.13 Notifications
------------------
Location: Dashboard header bell icon

Purpose: In-app alert center for staff.

How it works:
  - Loads notifications from repository
  - Unread count badge
  - Dropdown list with title and message
  - "Mark all read" calls db.notifications.markAllAsRead()
  - Types: BookingCreated, BookingCancelled, LeadCreated, System

Users: All authenticated staff
Business value: Real-time awareness of bookings and system events


2.14 Toast Notifications (UI Feedback)
----------------------------------------
Purpose: Transient user feedback for actions (success, error, warning, info).
Fixed bottom-right stack, auto-dismiss after 4 seconds, aria-live="polite".


2.15 Multi-Branch Support

Role Hierarchy

SECTION 4 — COMPLETE USER FLOWS
================================================================================

4.1 Patient Journey (Public)
----------------------------
  Landing Page (/)
      |
      +-- Browse services, doctors, FAQ, contact info
      |
      +-- "Book Appointment" -> /book/smilecare-clinic
              |
              Step 1: Select branch, service, doctor, date, time slot
              Step 2: Enter name, phone, email, optional notes
              Step 3: Choose payment (Cash / Stripe / PayPal) -> Confirm
              Step 4: Confirmation screen with reference number
              |
              -> Patient record created in system
              -> Appointment created (status: Pending)
              -> Staff notification generated

4.2 Receptionist Journey
------------------------
  /login -> Select "Receptionist" role -> Sign In
      -> Redirected to /admin/dashboard (branch scoped to br-main)
      -> Dashboard: Review today's schedule and KPIs
      -> Calendar: Create/edit/cancel appointments (conflict warnings)
      -> Patients: Search, register, open profiles, add notes
      -> Waitlist: Add patients, promote to appointments
      -> Leads: Add inquiries, advance pipeline stages
      -> Services / Doctors: View and manage (full CRUD access)
      -> Notifications: Review and mark alerts as read

4.3 Doctor / Staff Journey
--------------------------
  /login -> Select "Doctor / Staff" role -> Sign In
      -> Dashboard (branch fixed to br-main, no branch selector)
      -> View today's schedule and upcoming appointments
      -> Access calendar (can filter by staff including self)
      -> View patient profiles and history
      -> View services and doctor schedules
      -> Cannot access Reports or Settings

4.4 Clinic Owner / Administrator Journey
----------------------------------------
  /login -> Select "Clinic Owner" or "Super Admin" -> Sign In
      -> Full dashboard access with branch selector (all locations)
      -> All receptionist capabilities
      -> Reports: Analyze revenue, completion rates, export CSV/PDF
      -> Settings: Update clinic profile, currency, language (RTL)
                   Load/reload demo data or reset workspace
                   Toggle integration feature flags
                   Review audit activity log

4.5 Appointment Lifecycle (System Flow)
---------------------------------------
  Created (Pending)
      |
      +-- Confirmed -> Completed (+ payment tracking)
      +-- Cancelled
      +-- NoShow

  Payment parallel track: Unpaid -> Authorized -> Paid -> Refunded

6.3 Repository Pattern
----------------------
All data access flows through db (RepositoryProvider):
  db.branches | db.customers | db.reservations | db.services |
  db.employees | db.leads | db.notifications | db.waitlist |
  db.auditLogs | db.settings

Backend selection: VITE_USE_FIRESTORE === 'true'

6.4 Context Providers & State Flow
----------------------------------
  AuthContext  — currentUser, isLoading, login/logout, hasPermission()
  AppContext   — All domain entities, selectedBranchId, refreshData(),
                 audit logging
  ToastContext — Toast queue

Data flow:
  1. User logs in -> AuthContext persists session
  2. AppProvider calls refreshData() -> parallel repository fetches
  3. Components read from useApp() -> render UI
  4. Mutations call db.*.create/update/delete() -> refreshData()
  5. Significant actions call dispatchAuditLog()

6.5 Routing Architecture
------------------------
  - Lazy loading: All major page components use React.lazy() with Suspense
  - GuestOnlyRoute: redirects authenticated users away from / and /login
  - ProtectedAdminRoute: requires session; wraps DashboardLayout
  - RequireRole: enforces BusinessOwner+ for Reports and Settings
  - SPA fallback: Vercel rewrites all non-static paths to index.html

6.6 Firebase Architecture (Optional)
------------------------------------
  organizations/{orgId}/
    ├── branches/
    ├── patients/        (maps to Customer type)
    ├── appointments/    (maps to Reservation type)
    ├── services/
    ├── doctors/         (maps to Employee type)
    ├── leads/
    ├── notifications/
    ├── waitlist/
    ├── auditLogs/
    └── settings/        (document id: "main")

Firebase Auth is initialized in config.ts but NOT connected to the login flow.

6.7 Scalability Rationale
---------------------------
  - Repository interfaces: swap mock <-> Firestore without UI changes
  - Org-scoped Firestore paths: multi-tenant ready structure
  - Lazy-loaded routes: smaller initial bundle
  - Branch-scoped filtering: multi-location growth
  - Typed domain models: safe refactoring as features grow
  - Adapter pattern for integrations: plug in real Stripe/GCal/SMS later

Executive Summary
-----------------
SmileCare Pro is a modern dental clinic management platform delivered as a
React 19 single-page application with TypeScript, Tailwind CSS 4, and an
optional Firebase Firestore backend. It addresses the operational complexity
of running a dental practice by combining staff scheduling, patient records,
service and doctor management, waitlist handling, inquiry tracking, and
analytics into one responsive admin dashboard — alongside a public-facing
marketing site and online booking portal.

The application demonstrates production-oriented architecture through its
repository pattern (switchable between localStorage demo mode and Firestore),
role-based access control, bilingual English/Arabic support with RTL layout,
and Progressive Web App capabilities. It ships with a fully seeded dental
clinic demo dataset, enabling immediate evaluation without backend
configuration.

Built for portfolio demonstration and client proposals, the current release
simulates authentication, payments, and third-party integrations via adapter
stubs — providing a complete UX prototype ready for production hardening with
Firebase Auth, Firestore security rules, and real payment/notification providers.

GitHub README Summary
---------------------
SmileCare Pro — Enterprise dental clinic management SaaS. Manage appointments,
patients, doctors, services, waitlists, and patient inquiries from a unified
dashboard. Patients book online via a public portal. Built with React 19,
TypeScript, Tailwind CSS 4, and Firebase-ready architecture. Includes PWA
support, English/Arabic i18n, and portfolio demo mode.

Demo: /login (no password) | Booking: /book/smilecare-clinic

     Auth + security rules
  6. Extensibility: Adapter interfaces for payments, calendar, notifications ready
     for real SDK integration
